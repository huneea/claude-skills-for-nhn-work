# Create PR 스킬

현재 git 브랜치의 커밋들을 분석해서, 사용자가 지정한 타겟 브랜치로의 Pull Request를 `gh` CLI로 생성한다.

## 환경 판별

- **사내 GitHub (GHE)**: 리모트 호스트가 `github.nhnent.com`이거나, 사용자가 별도로 언급하지 않은 경우 기본값
- **공개 GitHub**: 리모트 호스트가 `github.com`이거나, 사용자가 "공개 repo", "public repo", "github.com" 등을 언급한 경우

## 사전 조건

- `gh` CLI가 설치되어 있고, GitHub (또는 GitHub Enterprise)에 인증이 완료된 상태여야 한다
- 현재 디렉토리가 git 저장소 안이어야 한다
- 작업 브랜치에 커밋이 있어야 한다 (타겟 브랜치 대비 새로운 커밋)

## 워크플로우

### 1단계: 현재 상태 파악

먼저 아래 정보를 수집한다:

```bash
# 현재 브랜치 이름
git branch --show-current

# 리모트 확인 (호스트로 사내/공개 판별)
git remote -v

# GitHub Enterprise인 경우 호스트 확인
gh auth status
```

리모트 URL의 호스트를 확인하여 사내/공개 GitHub를 자동 판별한다.

### 2단계: 필수 정보 확인

#### 사내 GitHub (GHE) 환경

아래 정보를 사용자에게 확인한다. 제공하지 않은 항목이 있다면 물어본다:

1. **타겟 브랜치** — "어떤 브랜치로 머지할까요? (예: develop, main, release/1.0 등)"
2. **두레이 TASK 태그명** — 먼저 현재 브랜치 이름에서 유추를 시도한다. 브랜치명에서 첫 번째 `/` 앞은 브랜치 타입(feature, hotfix 등)이고, 그 뒤가 두레이 TASK 태그명이다. 예를 들어 브랜치명이 `feature/AI-Care-Chat/107`이면 두레이 TASK 태그명은 `AI-Care-Chat/107`이다. 유추한 태그명을 사용자에게 확인받는다. 브랜치명에서 유추할 수 없는 경우에만 직접 물어본다. ← 이것은 **필수**이므로, 확인될 때까지 PR을 생성하지 않는다.
3. **관련 두레이 문서 링크** — "관련 두레이 문서 링크가 있으면 알려주세요. 없으면 없다고 해주셔도 됩니다." ← 필수는 아니지만, PR 생성 전에 반드시 한 번 확인한다. 사용자가 링크를 제공하면 PR 본문의 "관련 두레이 문서" 섹션에 포함하고, 없다고 하면 해당 섹션은 기본 안내 문구로 남겨둔다.

#### 공개 GitHub 환경

아래 정보만 확인한다:

1. **타겟 브랜치** — "어떤 브랜치로 머지할까요? (예: develop, main, release/1.0 등)"

두레이 TASK 태그명과 관련 문서 링크는 묻지 않는다.

### 3단계: 커밋 분석 및 PR 내용 작성

타겟 브랜치와 현재 브랜치의 차이를 분석한다:

```bash
# 타겟 브랜치 대비 새로운 커밋 목록
git log <target-branch>..HEAD --oneline --no-merges

# 변경된 파일 목록
git diff <target-branch>...HEAD --stat

# 상세 diff (내용 파악용)
git diff <target-branch>...HEAD
```

### PR 템플릿 — 사내 GitHub (GHE)

**제목 형식:**
```
#{두레이 TASK 태그명} - {PR 내용을 압축한 짧은 제목}
```

예시: `#PROJ-1234 - 사용자 인증 로직 리팩토링`

**본문 형식:**
```markdown
## 어떤 이유로 PR을 하셨나요?
- {커밋 내용과 diff를 분석하여 PR의 목적과 이유를 추론하여 작성}

## 작업 내용
- {PR에 포함된 커밋들의 작업 내용을 요약. 주요 변경사항별로 bullet point로 정리}

## 관련 두레이 문서
- 관련 두레이 문서가 있다면 링크를 첨부해주세요.

## 리뷰어가 집중해 주길 바라는 부분이 있다면 입력해 주세요.
- {diff 내용을 확인하여 리뷰어의 크로스체킹이 필요한 부분이나 공유할 만한 내용을 요약}

---
> 이 PR은 AI에 의해 작성되었습니다.
```

### PR 템플릿 — 공개 GitHub

**제목 형식:**
```
{PR 내용을 압축한 짧은 제목}
```

예시: `feat: 개인 프로젝트 조회 기능 추가`

**본문 형식:**
```markdown
## Summary
- {커밋 내용과 diff를 분석하여 PR의 목적과 변경 사항을 요약}

## Changes
- {PR에 포함된 커밋들의 작업 내용을 요약. 주요 변경사항별로 bullet point로 정리}

## Review Points
- {리뷰어가 특히 살펴봐야 할 포인트}

---
> Generated with AI
```

### 작성 요령 (공통)

- "이유/Summary" 섹션에는 단순히 "무엇을 했다"가 아니라, "왜 이 변경이 필요한지"를 커밋 메시지와 코드 변경 내용에서 추론하여 작성한다.
- "작업 내용/Changes"에는 커밋별로 나열하기보다, 논리적으로 묶어서 설명하는 것이 좋다.
- "리뷰 포인트"에는 새로 도입된 패턴, 기존 인터페이스 변경, 삭제된 기능 등 리뷰어가 특히 살펴봐야 할 포인트를 적는다.
- 사내 GHE는 한국어, 공개 GitHub는 영어로 작성한다 (사용자가 다른 언어를 요청하면 그에 따른다).

### 4단계: 사용자 확인

PR을 생성하기 전에 반드시 작성한 제목과 본문을 사용자에게 보여주고 확인을 받는다. 수정 요청이 있으면 반영한다.

### 5단계: 푸시 및 PR 생성

```bash
# 리모트에 현재 브랜치 푸시 (아직 안 되어 있다면)
git push -u origin <current-branch>

# PR 생성 (HEREDOC으로 본문 전달)
gh pr create --base <target-branch> --title "<제목>" --body "$(cat <<'EOF'
<본문 내용>
EOF
)"
```

GitHub Enterprise 환경에서는 `GH_HOST` 환경변수가 설정되어 있거나 `gh auth status`에서 올바른 호스트가 확인되어야 한다. 호스트 문제가 있으면 사용자에게 안내한다.

### 6단계: 결과 안내

PR이 생성되면 URL을 사용자에게 전달한다. 추가로 리뷰어 지정, 라벨 추가 등이 필요한지 물어본다.

## 주의사항

- `git push --force`는 절대 사용하지 않는다
- **사내 GHE 환경에서는 두레이 TASK 태그명 없이 PR을 생성하지 않는다** — 사용자가 제공할 때까지 반드시 물어본다
- **공개 GitHub 환경에서는 두레이 관련 항목을 묻지 않고, PR 템플릿에서도 제외한다**
- PR 생성 전에 반드시 사용자에게 제목과 설명을 보여주고 확인받는다
- 충돌(conflict)이 있으면 PR 생성 전에 사용자에게 알려준다
- 민감한 파일(.env, credentials 등)이 변경 사항에 포함되어 있으면 경고한다
