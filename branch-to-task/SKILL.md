# 브랜치 작업 내용 → 두레이 업무 문서 등록

현재 작업 브랜치의 커밋 히스토리를 분석하여 두레이 업무 문서로 등록하고, 등록된 업무 번호로 브랜치명을 변경한다.

## 워크플로우

### 1단계: 현재 브랜치 상태 확인

```bash
# 현재 브랜치명 확인
git branch --show-current

# base 브랜치(main, master, develop 등) 확인
git log --oneline --decorate --all | head -20

# 현재 브랜치의 커밋 히스토리 (base 브랜치 분기 이후)
git log --oneline <base-branch>..HEAD

# 변경된 파일 목록
git diff --stat <base-branch>..HEAD

# 상세 변경 내용
git diff <base-branch>..HEAD
```

- 현재 브랜치가 이미 `feature/{프로젝트코드}/{번호}` 형식이면 이미 업무 문서가 있을 수 있으므로 사용자에게 확인한다
- base 브랜치를 자동으로 판별한다 (main, master, develop 등)
- 커밋이 없으면 사용자에게 알리고 중단한다

### 2단계: 변경 내용 분석

커밋 히스토리와 diff를 분석하여 아래 내용을 정리한다:

- **업무 배경 (개요)**: 이 작업을 왜 하게 되었는지
- **작업 내용 (내용)**: 어떤 변경을 했는지 구조화하여 정리
  - 변경된 파일과 주요 변경 사항
  - 새로 추가된 기능, 수정된 버그, 리팩토링 등
- **참고**: 관련 파일 경로, 영향 범위 등

### 3단계: 프로젝트 확인

업무를 등록할 두레이 프로젝트를 사용자에게 확인한다.

- 브랜치명에 프로젝트 코드가 포함되어 있으면 이를 기본값으로 제안
- 아니면 사용자에게 프로젝트를 물어본다
- `dooray project list --search "<keyword>" --json` 으로 프로젝트 존재 여부를 확인한다

### 4단계: 업무 문서 작성 및 사용자 확인

`dooray-task` 스킬의 양식에 맞게 문서를 작성한다.

```
제목: [{프로젝트 코드}] {작업 내용 핵심 요약}
담당자: AI서비스개발팀-정훈희
```

```markdown
# 개요
- {분석된 업무 배경}

# 내용
- {분석된 작업 내용을 구조화하여 정리}

# 참고
- {변경된 주요 파일 경로, 영향 범위 등}
```

**반드시 사용자에게 작성된 문서를 보여주고 최종 확인을 받는다.**

### 5단계: 두레이 등록

`dooray-task` 스킬을 사용하여 업무를 등록한다.

```bash
cat <<'EOF' > /tmp/dooray-task-body.md
# 개요
- ...

# 내용
- ...

# 참고
- ...
EOF

dooray post create <project> \
  --subject "[{프로젝트코드}] {제목}" \
  --body-file /tmp/dooray-task-body.md \
  --to "정훈희" \
  --json

rm -f /tmp/dooray-task-body.md
```

등록 후 `post list`로 업무 번호를 확인한다.

### 6단계: 브랜치명 변경

등록된 업무 번호를 이용하여 현재 브랜치명을 변경한다.

```bash
# 현재 브랜치명을 feature/{프로젝트코드}/{업무번호} 로 변경
git branch -m <현재 브랜치명> feature/{프로젝트코드}/{업무번호}
```

- 이미 리모트에 push된 브랜치인 경우 사용자에게 알리고 리모트 브랜치 변경 여부를 확인한다
  ```bash
  # 리모트 브랜치도 변경할 경우
  git push origin --delete <이전 브랜치명>
  git push -u origin feature/{프로젝트코드}/{업무번호}
  ```

### 7단계: 결과 안내

완료 후 아래 정보를 사용자에게 알려준다:

- 등록된 업무 번호 및 taskNumber (예: `AI-Care-Chat/123`)
- 변경된 브랜치명 (예: `feature/AI-Care-Chat/123`)
- 리모트 동기화 여부

## 주의사항

- 커밋이 없는 브랜치에서는 실행하지 않는다
- 업무 문서 등록 전 반드시 사용자 최종 확인을 받는다
- 브랜치명 변경 전 리모트 push 상태를 확인한다
- `--body-file`을 사용하여 본문을 작성한다 (`--body`에 `\n`은 리터럴로 들어감)
- 업무 번호 확인은 `post list`를 사용한다 (`post search`는 인덱싱 지연 있음)
- 프로젝트 접근 실패 시 임의로 다른 프로젝트를 선택하지 않고 사용자에게 확인한다
