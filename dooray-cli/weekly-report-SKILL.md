# 주간 보고서 생성 스킬

두레이(Dooray)에서 업무 데이터를 수집하고, 프로젝트별로 분류하여 주간 보고서를 마크다운으로 생성한 뒤, 두레이에 등록한다.

## 의존성

- `dooray-cli` 스킬이 설치되어 있어야 한다
- dooray-cli가 설정 완료 상태여야 한다 (`dooray doctor`로 확인)

## 사용자 입력

사용자는 다음 정보를 제공한다:

| 항목 | 필수 | 설명 | 예시 |
|------|------|------|------|
| 등록 방식 | **필수** | 신규 업무 생성 또는 기존 업무 댓글 | 아래 참조 |
| 등록 대상 | **필수** | 방식에 따라 프로젝트명 또는 `{프로젝트명}/{업무번호}` | `AI서비스개발팀` 또는 `AI서비스개발팀/163` |
| 기간 | 선택 | 미지정 시 이번 주 월~오늘 | `4/7 ~ 4/11`, `지난주` |

## 실행 흐름

### 1단계: 등록 방식 및 대상 확인

사용자의 요청에서 등록 방식을 판별한다.

| 사용자 표현 | 등록 방식 |
|------------|----------|
| "~프로젝트에 작성", "~프로젝트에 등록", "업무 생성" | **신규 업무 생성** |
| "~업무에 댓글", "~에 등록", `프로젝트명/번호` 형식 제공 | **기존 업무 댓글** |

판별이 어려우면 사용자에게 물어본다:

> 주간 보고서를 어떻게 등록할까요?
> 1. **신규 업무 생성** — 두레이 프로젝트에 새 업무 문서로 등록 (프로젝트명 필요)
> 2. **기존 업무 댓글** — 기존 업무에 댓글로 등록 (프로젝트명/업무번호 필요)

**신규 업무 생성인 경우:**
- 프로젝트명을 확인한다. 미제공 시 물어본다.
- `dooray project list --search "<프로젝트명>" --json` 으로 프로젝트 존재 여부를 확인한다.

**기존 업무 댓글인 경우:**
- 프로젝트명과 업무번호를 분리한다.
- 입력: `AI서비스개발팀/163`
- 파싱: 프로젝트 = `AI서비스개발팀`, 업무번호 = `163`

### 2단계: 기간 확정

사용자가 기간을 명시하지 않으면 **이번 주 월요일 ~ 오늘**을 기본값으로 사용한다.

### 3단계: 프로젝트 목록 수집

내가 참여 중인 프로젝트 목록을 가져온다.

```bash
dooray project list --json
```

응답에서 프로젝트 `code`와 `name`을 추출한다. 이 목록이 보고서의 최상위 그룹이 된다.

### 4단계: 프로젝트별 업무 수집

각 프로젝트에 대해 업무 목록을 조회한다.

```bash
dooray post list <project-code> --json
```

조회된 업무 중 다음 조건에 해당하는 것만 필터링한다:
- 해당 기간 내에 **생성 또는 수정**된 업무
- **내가 담당자**인 업무 (JSON 응답의 `users.to[].member.name`으로 판별)

업무가 많은 프로젝트는 검색을 활용할 수도 있다:
```bash
dooray post search <project-code> "<keyword>" --json
```

### 5단계: 업무 상세 조회

필터링된 각 업무의 상세 내용을 가져온다.

```bash
dooray post get <project-code> <number> --json
```

상세에서 추출할 정보:
- `subject` → 업무명 / 세부사항
- `body` → 주요 작업 내용 파악에 활용
- 워크플로우 상태 → 진행 상태 (`[진행중]`, `[완료]` 등)

### 6단계: 보고서 구조화

수집한 데이터를 아래 계층으로 분류한다:

```
프로젝트 → 업무 카테고리(태그/상위업무) → 세부 업무 → 주요 내용
```

업무를 카테고리로 그룹핑할 때는:
- 업무의 태그, 상위 업무, 또는 제목의 공통 키워드를 기준으로 묶는다
- 적절한 카테고리가 없으면 업무 제목에서 핵심 도메인을 추출하여 그룹명으로 사용한다

### 7단계: 마크다운 생성

아래 양식을 **정확히** 따른다. 스타일 태그, 색상 코드, 헤딩 레벨을 그대로 유지할 것.

```markdown
## 프로젝트명
### <span style="color: rgb(32, 125, 229);">업무 카테고리</span>
##### <span style="color: rgb(0, 107, 117);">세부 업무명</span> <span style="color: red;">[상태]</span>
* 주요 내용 요약
    * 세부 내용
* 주요 내용 요약
    * 세부 내용
```

**상태 표기 규칙:**
- 완료된 업무 → `[완료]`
- 진행 중인 업무 → `[진행중]`
- 등록만 된 업무 → `[등록]`

**작성 스타일:**
- 문장을 길게 쓰지 않는다. 간결한 구(phrase) 또는 짧은 문장으로 작성한다
- "~을 하였음", "~을 진행함" 같은 종결어미 대신 "~작업", "~변경", "~추가" 같은 명사형으로 마무리한다
- 핵심 정보(변경 대상, 수치, 기술 키워드)를 앞에 배치한다
- 불필요한 조사와 접속사를 최소화한다

### 8단계: 사용자 확인

생성된 보고서를 사용자에게 보여주고 확인을 받는다.
- 수정 요청이 있으면 반영한다
- 확인이 되면 다음 단계로 진행한다

### 9단계: 두레이 등록

1단계에서 결정된 등록 방식에 따라 진행한다.

#### 방식 A: 신규 업무 생성

보고서를 새 업무 문서로 생성한다. 제목 형식: `[주간업무보고] YYYY-MM-DD ~ YYYY-MM-DD`

```bash
cat <<'EOF' > /tmp/weekly-report.md
(보고서 마크다운 내용)
EOF

dooray post create <프로젝트명> \
  --subject "[주간업무보고] 2026-04-13 ~ 2026-04-17" \
  --body-file /tmp/weekly-report.md \
  --to "정훈희" \
  --json

rm -f /tmp/weekly-report.md
```

등록 성공 시 `post list`로 업무 번호를 확인하여 사용자에게 알려준다.

#### 방식 B: 기존 업무 댓글

확인된 보고서를 지정된 업무에 댓글로 작성한다.

```bash
dooray post comment add <프로젝트명> <업무번호> --body-file ./weekly-report.md
```

예시:
```bash
dooray post comment add AI서비스개발팀 163 --body-file ./weekly-report.md
```

---

등록이 성공하면 사용자에게 완료를 알린다.
등록에 실패하면 에러 내용을 알리고, 마크다운 파일을 별도로 제공한다.

## 작성 예시

아래는 실제 주간 보고서 작성 예시이다. 이 수준의 간결함과 구조를 유지할 것.

```markdown
## AI-TF-VectorSearch
### <span style="color: rgb(32, 125, 229);">API</span>
##### <span style="color: rgb(0, 107, 117);">검색 API 분석 및 리팩토링</span> <span style="color: red;">[완료]</span>
* 현재 검색 API 동작 흐름 파악
* 가독성 향상을 위한 코드 리팩토링
##### <span style="color: rgb(0, 107, 117);">공통/그룹 인덱스 구조 변경에 따른 대응</span> <span style="color: red;">[진행중]</span>
* 검색 API 에서 검색 대상을 선별하는 기준 변경
* 공통과 그룹 인덱스들의 소스가 모두 물리적으로 분리되는 구조로 일단 진행 중 (기획에 따라 변경 가능)
    * 변경대상 : 검색API, 타겟별 임베딩 (저장) API
### <span style="color: rgb(32, 125, 229);">BATCH</span>
##### <span style="color: rgb(0, 107, 117);">두레이 TASK 색인시 참조자 정보 추가</span> <span style="color: red;">[진행중]</span>
* LLM 팀 요청으로 두레이 TASK 색인시 참조자 정보를 임베딩시 포함하도록 변경
* TASK 변경 여부 확인 기준 변경
    * as-is : 수정일시, 본문 내용
    * to-be : 수정일시, 본문 내용, 담당자, 참조자
* TASK 전체 재색인 필요함
## Face-Recognition
### <span style="color: rgb(32, 125, 229);">보안</span>
##### <span style="color: rgb(0, 107, 117);">[2026] NHN 위험평가 - 서버(Linux) 취약점 점검</span> <span style="color: red;">[완료]</span>
* 점검 내용 : `수동점검 자산 (158대)` 의`face-recognition` 서비스 관련 서버들의 취약점 점검 스크립트 수동 실행
* 대상 서버 :
    * face-recognition-beta-application-node-0
    * face-recognition-beta-application-node-1
```

## 주의사항

- 업무가 없는 프로젝트는 보고서에 포함하지 않는다
- 프로젝트 간 구분은 `##` 헤딩으로 한다
- 보고서 맨 위에 제목이나 날짜 헤더를 별도로 추가하지 않는다 (사용자가 요청하면 추가)
- dooray-cli 호출 시 항상 `--json` 플래그를 사용하여 구조화된 데이터를 받는다
- API 호출이 실패하면 에러를 사용자에게 알리고, 가능한 프로젝트만으로 보고서를 생성한다
- 댓글 등록 전 반드시 사용자 확인을 받는다 — 자동으로 바로 등록하지 않는다
- 최종 보고서는 `.md` 파일로도 로컬에 저장하여 백업을 유지한다
